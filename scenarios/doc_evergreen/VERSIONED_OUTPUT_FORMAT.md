# Versioned Output Format

This document describes the standardized format for all versioned output JSON files in doc-evergreen.

## Directory Structure

Versioned outputs are stored in numbered directories that indicate their step in the workflow:

- **Step 2**: `.doc-evergreen/2_file_summaries/` - File summarization outputs
- **Step 3**: `.doc-evergreen/projects/{project_name}/3_relevancy/` - Relevancy scoring outputs
- **Step 4**: `.doc-evergreen/projects/{project_name}/4_customized_template/` - Customized template outputs
- **Step 5**: `.doc-evergreen/projects/{project_name}/5_generated_content/` - Generated document outputs

## Standard Structure

All versioned outputs follow this structure:

```json
{
  "version": "2025-01-14T12:34:56.789Z",
  "template_version_id": "customized-20250114_123456",
  "prompt": {
    "name": "prompt_name",
    "version": "2025-01-14T00:00:00Z"
  },
  "inputs": {
    "input_field_1": "value1",
    "input_field_2": "value2"
  },
  "outputs": {
    "output_field_1": "result1",
    "output_field_2": "result2"
  }
}
```

## Fields

### Required Fields

- **`version`**: ISO 8601 timestamp when this version was created
- **`outputs`**: Dictionary containing all outputs generated by the operation

### Optional Fields

- **`template_version_id`**: Human-readable identifier for this version (optional)
- **`prompt`**: Information about the LLM prompt used (if applicable)
  - **`name`**: Name of the prompt file (e.g., "create_customized_template")
  - **`version`**: Version timestamp of the prompt used
- **`inputs`**: Dictionary containing all inputs that were used to generate the outputs

## Examples

### Customized Template

```json
{
  "version": "2025-01-14T12:34:56.789Z",
  "template_version_id": "customized-20250114_123456",
  "prompt": {
    "name": "create_customized_template",
    "version": "2025-01-14T00:00:00Z"
  },
  "inputs": {
    "doc_description": "API documentation for the auth service",
    "template_guide_version": "2025-01-13T23:00:00Z",
    "source_files": [
      {
        "file_path": "src/auth.py",
        "summary": {"text": "Authentication module", "timestamp": "2025-01-13T20:00:00Z"},
        "relevancy": {
          "explanation": "Core auth logic",
          "score": 9,
          "prompt": {"name": "score_relevancy", "version": "2025-01-13T23:00:00Z"}
        }
      }
    ]
  },
  "outputs": {
    "customized_template": "# API Documentation\n\n## Authentication\n...",
    "selected_files": {
      "src/auth.py": "Needed for authentication section"
    }
  }
}
```

### File Summary

File summaries now use the standardized format:

**File**: `.doc-evergreen/2_file_summaries/{sanitized_filename}.json`

```json
{
  "name": "src__auth__py",
  "versions": [
    {
      "version": "2025-01-14T12:34:56.789Z",
      "outputs": {
        "summary": "Authentication module implementing OAuth2 flow"
      },
      "prompt": {
        "name": "prompts/summarize_file.json",
        "version": "2025-01-14T00:00:00Z"
      },
      "inputs": {
        "file_path": "src/auth.py",
        "commit_hash": "abc123def456"
      }
    }
  ]
}
```

**Key features**:
- Top-level `name` field uses sanitized filename (not `file_path`)
- `version` field contains ISO 8601 timestamp for this version
- Order: version first, then outputs, then prompt, then inputs
- Prompt name is full file path (e.g., `prompts/summarize_file.json`)
- Multiple versions tracked over time (when file changes)
- Each version includes commit hash for git tracking
- Legacy formats automatically migrated on next write

### Relevancy Scores (Step 3)

Relevancy scores follow the same standardized format:

**File**: `.doc-evergreen/projects/{project_name}/3_relevancy/{sanitized_filename}.json`

```json
{
  "project_name": "api-docs",
  "name": "src__auth__py",
  "versions": [
    {
      "version": "2025-01-14T13:00:00.000Z",
      "outputs": {
        "relevancy": {
          "explanation": "This file contains the core authentication logic needed for the API documentation",
          "score": 9
        }
      },
      "prompt": {
        "name": "prompts/score_relevancy.json",
        "version": "2025-01-14T00:00:00Z"
      },
      "inputs": {
        "doc_description": "API documentation for the authentication module",
        "file_summary": {
          "name": "2_file_summaries/src__auth__py.json",
          "version": "2025-01-14T12:34:56.789Z"
        }
      }
    }
  ]
}
```

**Key features**:
- Top-level `project_name` identifies which document this relevancy is for
- Top-level `name` field uses sanitized filename
- `version` field contains ISO 8601 timestamp for this version
- Order: version first, then outputs, then prompt, then inputs
- Outputs contain the full LLM output as a `relevancy` object (not broken apart)
- Inputs reference the document description and file summary used
- File summary input includes both name (path to summary file) and version (timestamp)
- Multiple versions tracked as relevancy is re-scored

### Generated Content (Step 5)

Generated document content follows the same standardized format:

**File**: `.doc-evergreen/projects/{project_name}/5_generated_content/{document_name}.json`

```json
{
  "name": "API_Documentation",
  "versions": [
    {
      "version": "2025-01-14T14:00:00.000Z",
      "outputs": {
        "document_content": "# API Documentation\n\n## Authentication\n..."
      },
      "prompt": {
        "name": "prompts/generate_document.json",
        "version": "2025-01-14T00:00:00Z"
      },
      "inputs": {
        "customized_template": "4_customized_template/customized_template_20250114_120000.json",
        "customized_template_version": "2025-01-14T12:00:00.000Z",
        "source_files": [
          "src/auth.py",
          "src/api.py"
        ]
      }
    }
  ]
}
```

**Key features**:
- Top-level `name` field identifies the document
- `version` field contains ISO 8601 timestamp for this version
- Order: version first, then outputs, then prompt, then inputs
- Prompt name is full file path
- Inputs reference the customized template and source files used
- Multiple versions tracked as document is regenerated

## Benefits

1. **Clear separation**: Inputs vs outputs are explicitly labeled
2. **Traceability**: Prompt name and version track what LLM operation was used
3. **Reproducibility**: All inputs are stored for potential regeneration
4. **Consistency**: Same structure across all versioned outputs
5. **Debugging**: Easy to see exactly what went in and what came out
